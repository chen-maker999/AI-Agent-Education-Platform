# RAG系统架构文档

## 1. 系统概述

本RAG系统采用本地轻量级技术栈，结合语义检索和关键词检索，为用户提供准确、高效的信息检索和生成服务。系统设计专注于检索链和离线索引优化，确保在有限资源下实现最佳性能。

## 3. 技术选型

### 3.1 嵌入模型

- **技术**：本地轻量级TF-IDF + jieba分词
- **优势**：
  - 无需依赖外部模型，部署简单
  - 支持中文文本处理
  - 动态计算向量维度，适应不同规模的数据集
  - 计算资源需求低，响应速度快

### 3.2 向量数据库

- **技术**：FAISS (Facebook AI Similarity Search)
- **优势**：
  - 高性能向量检索
  - 支持多种索引类型
  - 内存占用可控
  - 适合中小规模数据集
  - 易于集成和部署

### 3.3 关键词检索

- **技术**：BM25算法
- **优势**：
  - 传统信息检索领域的经典算法
  - 对关键词匹配效果好
  - 计算效率高
  - 与语义检索互补

### 3.4 生成模型

- **技术**：Kimi大语言模型
- **优势**：
  - 中文理解能力强
  - 上下文处理能力优秀
  - 支持长文本生成
  - API调用方式简单

### 3.5 后端框架

- **技术**：Flask
- **优势**：
  - 轻量级Web框架
  - 易于部署
  - 适合构建RESTful API
  - 开发效率高

## 4. 检索链设计

### 4.1 查询处理

1. **查询分析**：分析用户查询类型和意图
2. **查询重写**：优化查询表达，提高检索效果
3. **查询扩展**：扩展关键词，增加检索覆盖面

### 4.2 双重检索策略

1. **语义检索**：
   - 使用TF-IDF向量计算语义相似度
   - 检索Top-K语义相关文档
2. **关键词检索**：
   - 使用BM25算法计算关键词匹配度
   - 检索Top-K关键词相关文档
3. **结果融合**：
   - 去重处理
   - 分数融合
   - 重排序

### 4.3 上下文构建

1. **文档选择**：选择最相关的文档片段
2. **上下文组织**：构建结构化的上下文
3. **长度控制**：确保上下文不超过模型输入限制

### 4.4 生成与后处理

1. **提示构建**：设计有效的提示模板
2. **回答生成**：基于上下文生成回答
3. **结果验证**：确保回答基于检索结果
4. **格式处理**：调整回答格式和结构

## 5. 离线索引优化

### 5.1 数据处理优化

1. **文本分段策略**：
   - 基于段落的分段
   - 控制分段大小（默认500-1000字符）
   - 保留段落边界，确保语义完整性
2. **元数据提取**：
   - 提取文档标题、作者、日期等元数据
   - 为每个文档片段添加元数据标签
   - 支持基于元数据的过滤

### 5.2 索引构建优化

1. **批处理**：
   - 批量处理文档嵌入
   - 减少I/O操作
   - 提高索引构建速度
2. **索引类型选择**：
   - 对于中小规模数据集，使用IndexFlatIP
   - 对于大规模数据集，考虑IndexIVFFlat
3. **索引保存与加载**：
   - 将索引保存到磁盘
   - 支持快速加载
   - 减少系统启动时间

### 5.3 检索性能优化

1. **缓存策略**：
   - 缓存频繁查询的结果
   - 缓存热门文档的向量
2. **并行处理**：
   - 并行执行语义检索和关键词检索
   - 提高检索速度
3. **查询优化**：
   - 优化TF-IDF计算
   - 减少检索时的计算开销

## 6. 系统流程

### 6.1 索引构建流程

1. **数据收集**：从知识源获取原始数据
2. **数据处理**：清洗、分段、提取元数据
3. **向量嵌入**：使用TF-IDF + jieba分词生成向量
4. **索引构建**：使用FAISS构建向量索引
5. **索引保存**：将索引保存到磁盘

### 6.2 查询处理流程

1. **接收查询**：获取用户输入的查询
2. **查询优化**：重写、扩展查询
3. **双重检索**：执行语义检索和关键词检索
4. **结果融合**：合并、去重、排序
5. **上下文构建**：组织检索结果为上下文
6. **生成回答**：使用Kimi生成回答
7. **结果处理**：验证、格式化回答
8. **返回结果**：将回答返回给用户
